## 一、代码审查发现

### 🔴 P0 — 真实 Bug (已修复)

#### 1. `/api/test-all` 硬编码 Linux 路径 (已修)
- **文件**: `src/app/api/test-all/route.ts:8`
- **问题**: `const UPLOAD_DIR = '/home/z/my-project/uploads';` 写死 Linux 路径,在 macOS 上执行会失败。
- **修复**: 改为尊重 `process.env.SCRIPT_MANAGER_UPLOAD_DIR`,fallback 到 `process.cwd()/uploads`,与 `/api/execute` 保持一致。
- **影响**: 如果不修,此端点会因 path 不存在而走 `catch` 分支,`mkdir` 会在错误位置建目录并把 mock 文件写到无关位置,导致下游 execute 找不到输入文件。

#### 2. `/api/test-all` 端口写死 3003 (已修)
- **文件**: `src/app/api/test-all/route.ts:243`
- **问题**: `process.env.PORT || 3003`,但 `package.json` 实际用 3002 (且 playwright config 期望 3002)。
- **修复**: 改为 `process.env.PORT || '3002'`。
- **影响**: 不修的话,test-all 内部 fetch 永远连不上自己,会全部返回 `API returned 500`。

### 🟡 P1 — 参数解析 Bug (待修)

#### 3. 含 `/` 的复合 param 名被原样作为 flag 传给 argparse
- **文件**: `src/app/api/execute/route.ts:271-279` (argparse 分支)
- **现象**:
  - 测试结果中 `rename chainid` 错误信息:
    ```
    unrecognized arguments: test test /tmp/script-manager/xxx_test.pdb
    ```
  - 实际传入:`["--"-i / --input", "test", "-o / --output", "test", "--n_per_layer_protein", "4", ...]`
- **根因**: `seed-local` 维护的 param metadata 里把 flag 写成 `"-i / --input"` 这种"展示名",被 schema 序列化进了 `params` JSON,executor 看到 key 不以 `-` 开头就用 `--{key}` 前缀,但 key 中含空格导致整个串被吃成一个 flag。
- **修复方向**:
  1. **短期**:在 `extractParams` (lib/script-analyzer.ts) 里增加一个 sanitize 步骤,把 `name` 中所有非 `[\w-]` 字符替换成 `_`,或者拆出第一个 `--xxx` 模式作为 name。
  2. **长期**:重新跑一次 `seed-local` 把这些损坏的 param schema 清掉(手动编辑),或者在 `POST /api/scripts` 时校验 param name 格式。
- **影响**: 影响 4 个脚本 (`add scalebar`, `ppt to images`, `rename chainid`, `rename chainid2`) 在 web 中能"看到参数"但"运行失败"。

#### 4. argparse + positional args 同时存在时参数重复追加
- **文件**: `src/app/api/execute/route.ts:270-281`
- **现象**: `add scalebar` 错误:
  ```
  error: unrecognized arguments: --directory /tmp/script-manager/xxx_test.txt /tmp/script-manager/xxx_test.txt
  ```
  (同一个 input file 被追加了两次)
- **根因**: 第 280 行 `for (const fp of inputFileList) args.push(fp);` 没有判断"这个 input 是否已经被 `params.directory` 覆盖"。mock param 中 `directory=.` + mockInputFiles 中有 `{magnification}_{index}.tiff` → 两者都会被加进 args。
- **影响**:argparse + 有 input file 装饰器模式的脚本,会参数串错位。

### 🟡 P2 — 设计/一致性

#### 5. `executeCommand` 内的 `proc.on('close')` 与 `proc.on('error')` 都用 `resolved` flag 防止双 resolve,但 `setTimeout` 的 timeout 路径没有清理
- **文件**: `src/app/api/execute/route.ts:38-94`
- **问题**: spawn 自带的 `timeout` 选项会发送 SIGTERM,但这里没有 SIGKILL 兜底;如果 Python 进程忽略 SIGTERM,proc 永远不 close,ExecutionLog 永远停留在 `running` 状态(虽然 status 计算为 `timeout`)。
- **影响**: 实际表现为 — 3 个 timeout 的脚本 (`ANARCI`, `cli runner`, `run script`) 的 `exitCode: null`、`status: 'error'`(应该是 `timeout`),但前端展示无问题。
- **建议**: 用 `tree-kill` 库或者设置 `killSignal: 'SIGKILL'`,并把 execId 写到 ExecutionLog 的 `error` 字段里"Timeout after Xms"。

#### 6. `/api/execute` 路径遍历防御 OK,但 `resultFiles` 复制未做 size 限制
- **文件**: `src/app/api/execute/route.ts:338-358`
- **问题**: `readdir(TMP_DIR)` 收集所有 `runId` 前缀的文件,如果脚本在 /tmp 写了 10GB 临时文件,会被全量复制到 `UPLOAD_DIR`(同样是持久化 SQLite 的 db 所在机器)。DoS / 磁盘耗尽风险。
- **建议**: 加一个 `MAX_RESULT_FILE_SIZE = 50 * 1024 * 1024` 之类的限制 + 总大小 N 个文件限制。

#### 7. `seed-local` 维护的 33 个 script metadata 中 param schema 有 4 个损坏(P1 关联)
- **文件**: `src/app/api/seed-local/route.ts:11-450` 内的 `SCRIPT_REGISTRY`
- **现象**: param name 含 `/`、空格、`{xxx}` 占位符
- **建议**: 加一个 `validateScriptMeta()` 工具函数,在 `POST` 时拒绝入库。

#### 8. `UPLOAD_DIR` 在 `seed/route.ts` 里走 `fetch` GitHub 而 `seed-local/route.ts` 走 `readFileSync`,语义不一致但没有注释说明
- **影响**: 低,纯代码可读性。

#### 9. `extractDescription` 中 "keywordMap" 只取第一个匹配的 kw
- **文件**: `src/lib/script-analyzer.ts:64-89`
- **问题**: 例如 `epitope_visualizer_pptx.py`,name 含 `epitope` 和 `pptx`,只能匹配到 `epitope`,不会追加 `pptx` 的描述。
- **影响**: 低,只是展示问题。
